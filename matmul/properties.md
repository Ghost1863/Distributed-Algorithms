# LTL-свойства модели

Модель использует две **observable**-переменные (только для верификации,
алгоритм их для коммуникации не использует):

| Переменная | Где пишется | Смысл                                                    |
|------------|-------------|----------------------------------------------------------|
| `k`        | менеджер    | число принятых столбцов-результатов, `0..N`              |
| `done`     | менеджер    | `1` ровно после рассылки `STOP` всем работникам          |

Операторы LTL: `[]` — globally (всегда), `<>` — eventually (когда-нибудь),
`->` — импликация.

---

## Безопасность (safety) — «плохого никогда не случится»

### `safety_bounded`

```promela
ltl safety_bounded { [] (k <= N) }
```

**Счётчик принятых столбцов никогда не превышает `N`.** Менеджер ждёт
ровно `N` ответов, поэтому `k ∈ {0..N}` инвариантно.

### `done_after_all`

```promela
ltl done_after_all { [] (done == 1 -> k == N) }
```

**`done = 1` только после сборки всех `N` столбцов.** Нарушение
означало бы преждевременное завершение.

### `cols_stable`

```promela
ltl cols_stable { [] (k == N -> [] (k == N)) }
```

**`k` монотонен:** достигнув `N`, не убывает. Это фундаментальнее
стабильности `done`, поскольку `done` — производное от `k`.

---

## Живость (liveness) — «хорошее рано или поздно наступит»

### `liveness_done`

```promela
ltl liveness_done { <> (done == 1) }
```

**Алгоритм всегда завершается.** Нарушение означало бы взаимную
блокировку или голодание.

### `progress_collect`

```promela
ltl progress_collect { [] (k < N -> <> (k == N)) }
```

**Прогресс гарантирован из любого промежуточного состояния**, не
только из начального. Строже простой живости: даже при частично
собранном `C` оставшиеся столбцы будут собраны.

### `cols_then_done`

```promela
ltl cols_then_done { [] (k == N -> <> (done == 1)) }
```

**Как только все столбцы собраны, менеджер обязательно дойдёт до
`done = 1`** (разошлёт `STOP` и завершится). «Обратная» сторона
`done_after_all`: первое говорит «`done` ⇒ всё собрано», это —
«всё собрано ⇒ когда-нибудь `done`».

## Сводка

| #  | Свойство            | Тип                  | Формула                                |
|----|---------------------|----------------------|----------------------------------------|
| 1  | `safety_bounded`    | safety               | `[] (k <= N)`                          |
| 2  | `done_after_all`    | safety (impl.)       | `[] (done == 1 -> k == N)`             |
| 3  | `cols_stable`       | safety (stability)   | `[] (k == N -> [] (k == N))`           |
| 4  | `liveness_done`     | liveness             | `<> (done == 1)`                       |
| 5  | `progress_collect`  | conditional liveness | `[] (k < N -> <> (k == N))`            |
| 6  | `cols_then_done`    | conditional liveness | `[] (k == N -> <> (done == 1))`        |

## Соответствие теоремам учебника (раздел 6.4)

| Теорема книги                              | Покрыто         |
|--------------------------------------------|-----------------|
| **Т.2** — завершаемость                    | `liveness_done`, `cols_then_done` |
| **Т.3** — отсутствие тупиков               | автоматически Spin'ом (`+ invalid endstates`) |
| **Т.1.1–1.7** — внутренние инварианты      | неявно покрыто полным обходом графа |
| **Т.1.8** (`|z| = k`)                      | `safety_bounded`, `cols_stable`, `done_after_all` |
| **Т.1.9** (`∀ j: C_j = A_j · B`)           | LTL'ом не выражается; добавляется `assert` (см. README) |

## Запуск проверки всех 6 свойств

```bash
spin -a matmul.pml && gcc -O2 -o pan pan.c
for p in safety_bounded done_after_all cols_stable \
         liveness_done progress_collect cols_then_done; do
  echo "=== $p ==="
  ./pan -a -f -N $p | grep -E "errors:|completed|violated"
done
```

Флаг `-f` включает weak fairness — обязателен для liveness-свойств
(`liveness_done`, `progress_collect`, `cols_then_done`).
