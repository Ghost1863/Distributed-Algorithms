# Параллельное умножение матриц на Promela / Spin

Модель параллельного умножения двух матриц `N × N` силами `P`
процессов с распределением работы **по столбцам**. Реализация по
схеме «менеджер–работники» из учебника (раздел 6.4), адаптированной
к столбцам:

- разные работники одновременно вычисляют разные столбцы `C`;
- **никакой разделяемой памяти** между процессами для исходных
  данных (`A`, `B` локальны у каждого работника);
- обмен через `P + 1` каналов;
- менеджер (`init` / `P_0`) выводит итоговую матрицу `C` — а заодно
  и `A`, `B`.

В каталоге два файла:

- [matmul2.pml](matmul2.pml) — модель Promela + 4 LTL-свойства.
- [README.md](README.md) — этот файл.

---

## 1. Что такое Spin и Promela

**Spin** — model-checker для конечных распределённых систем. Принимает
модель на языке **Promela** (PROcess MEta-LAnguage) и:

1. **Симулирует** её (random / interactive / guided) — одно случайное
   выполнение для проверки «как это вообще работает».
2. **Верифицирует** её — генерирует C-программу `pan.c`
   (**P**rocess **AN**alyser), которая компилируется в `pan` и
   **полностью обходит** пространство состояний модели, проверяя:
   - инварианты (`assert`);
   - тупики (`invalid endstates`);
   - LTL-свойства (temporal logic);
   - циклы прогресса / acceptance.

**iSpin** — Tk-GUI поверх `spin` и `gcc`. Все его кнопки в итоге
вызывают `spin` → `gcc` → `./pan` и пишут результат в правую панель
и в `pan.out`.

## 2. Идея алгоритма (по схеме учебника)

- **Менеджер `P_0`** (`init`) и **`P` работников** (`Worker`).
- У каждого работника **свои локальные копии** `A`, `B` (никакой
  shared memory; матрицы инициализируются процессом самостоятельно
  через inline `init_AB`).
- **Адресные каналы** `task[P] = [1] of {byte}`: `task[i]` — труба от
  менеджера строго к работнику `i`. По ней передаётся либо номер
  столбца `j ∈ {1..N}`, либо `STOP = 0`.
- **Общий канал** `result = [P] of {byte, byte, Col}`: любой работник
  возвращает `(id, j, столбец)` менеджеру.
- Протокол менеджера повторяет диаграмму состояний из книги:
  - `0→0`: первичная раздача `1..min(P, N)`;
  - `1↔2`: приём результата, выдача следующей задачи тому же работнику
    (динамическая балансировка);
  - `3→3`: рассылка `STOP` всем `P` работникам.

### Почему два потока — два разных канала, а не один

- адресная посылка задач конкретному работнику требует отдельной
  трубы к каждому из них (`P` труб);
- сборка ответов — естественный fan-in от `P` работников к одному
  менеджеру (одна общая труба);
- совмещать оба потока в одной трубе можно, но пришлось бы
  тегировать сообщения и фильтровать «своё/не своё» — лишняя
  сложность без выигрыша.

## 3. LTL-свойства

```
ltl safety_bounded   { [] (ready_cols <= N) }
ltl liveness_done    { <> (done == 1) }
ltl done_after_all   { [] (done == 1 -> ready_cols == N) }
ltl progress_collect { [] (ready_cols < N -> <> (ready_cols == N)) }
```

| Свойство            | Тип                  | Что гарантирует                                                    |
|---------------------|----------------------|--------------------------------------------------------------------|
| `safety_bounded`    | safety               | Счётчик собранных столбцов никогда не превышает `N`.               |
| `liveness_done`     | liveness             | Алгоритм когда-нибудь завершается (`done = 1`).                    |
| `done_after_all`    | safety (impl.)       | Завершение возможно только когда все `N` столбцов собраны.         |
| `progress_collect`  | conditional liveness | Если ещё не все собраны — когда-нибудь будут собраны все.          |

Операторы LTL: `[]` — globally, `<>` — eventually, `->` — импликация.

`done` и `ready_cols` — глобальные observable-переменные **только для
LTL**. Алгоритм их для коммуникации не использует; пишет в них
исключительно менеджер.

### Связь с теоремами учебника

| Теорема книги (раздел 6.4)              | Что у нас                                                           |
|-----------------------------------------|---------------------------------------------------------------------|
| **Т.2** — завершаемость                 | ✅ `liveness_done`                                                 |
| **Т.3** — нет тупиков                   | ✅ автоматически Spin'ом (`+ invalid endstates`)                   |
| **Т.1.1–1.7** — внутренние инварианты   | ✅ неявно покрыто полным обходом графа Spin'ом                     |
| **Т.1.8** (`|z| = k`)                   | ✅ `safety_bounded` + `done_after_all`                             |
| **Т.1.9** (`∀ j ∈ z: C_j = A_j · B`)    | ⚠️ арифметическая корректность LTL'ом не проверяется (см. ниже)     |

> Теорема 1.9 — корректность арифметики результата — не выражается
> через LTL без громоздких квантификаторов. Если нужно проверить
> формально, добавь `assert` в менеджер после раскладки столбца:
>
> ```promela
> /* в init, после "C[i*N + (rj-1)] = col.v[i];" */
> byte exp, kk2;
> exp = 0; kk2 = 0;
> do
> :: (kk2 < N) -> exp = exp + A[i*N + kk2] * B[kk2*N + (rj-1)]; kk2++
> :: (kk2 == N) -> break
> od;
> assert(C[i*N + (rj-1)] == exp);
> ```

## 5. Запуск через iSpin GUI

`ispin matmul2.pml`. Сверху — вкладки.

### 5.1. Edit/View
Просмотр/правка `matmul2.pml`. **Save** обязательна перед любым
другим действием — Spin читает файл с диска.

### 5.2. Syntax Check
Запускает `spin -a` и парсит вывод. Печать LTL-формул и сообщение
«the model contains 4 never claims … only one claim is used in a
verification run» — это **информация**, не ошибка.

### 5.3. Simulate / Replay
- **Mode**:
  - `Random, with seed` — случайная симуляция (детерминированная для
    одного и того же seed).
  - `Interactive` — на каждом недетерминированном шаге GUI спрашивает.
  - `Guided, with trail` — воспроизводит контрпример из `.trail`-файла.
    Если trail отсутствует или сломан — будет ошибка
    «trail ends after −N steps» / «couldn't find claim 1». Лечится
    удалением `matmul2.pml.trail` или переключением в Random.
- **A Full Channel** — что делать при попытке записать в полный канал
  (у нас не возникает).
- **Output Filtering** — regex-фильтр для `printf`.
- **Track Data Values** — показывать значения переменных слева внизу.
- **MSC** справа — Message Sequence Chart, видно потоки сообщений
  между процессами.

> Сообщения `couldn't open "Worker": no such file or directory` —
> GUI ищет отдельный файл для proctype. У нас всё в одном `.pml`,
> поэтому сообщения **безвредны**.

### 5.4. LTL Property Manager
Можно редактировать LTL прямо в GUI. У нас они уже встроены в `.pml`,
так что вкладка не нужна.

### 5.5. Verification — ключевая вкладка

Правильные настройки для нашей задачи:

**Safety**:
- ☑ `+ invalid endstates (deadlock)`
- ☑ `+ assertion violations`

**Liveness**:
- ⦿ `acceptance cycles`
- ☑ **`enforce weak fairness constraint`** ← обязательно для liveness

**Storage Mode**:
- ⦿ `exhaustive`
- При больших `N`: ☑ `+ collapse compression`

**Search Mode**:
- ⦿ `depth-first search`
- ☑ `+ partial order reduction`

**Never Claims**:
- ⦿ `use claim`
- В поле **`claim name (opt):`** — имя одного из четырёх свойств:
  `safety_bounded`, `liveness_done`, `done_after_all`,
  `progress_collect`. Каждое запускать **отдельно**.

→ нажать **Run**.

Признак успеха в правой панели/`pan.out`:
```
errors: 0
... states, stored
(без "Warning: Search not completed")
```

## 6. Запуск из командной строки

Все команды — из каталога модели:

```bash
cd /home/immortal/spin/matmul
```

### 6.1. Симуляция

```bash
spin matmul2.pml
```

Печать матрицы `A`, `B = I` и `C = A·B = A` подтверждает корректность
(при инициализации по умолчанию).

### 6.2. Верификация LTL

```bash
spin -a matmul2.pml
gcc -O2 -o pan pan.c

for p in safety_bounded liveness_done done_after_all progress_collect; do
  echo "=== $p ==="
  ./pan -a -f -N $p | grep -E "errors:|completed|violated"
done
```

Флаг `-f` включает weak fairness (нужен для liveness).

### 6.3. Если свойство нарушено

`pan` запишет `matmul2.pml.trail`. Воспроизвести:

```bash
spin -t -p -g -l matmul2.pml
```

В GUI это `Simulate/Replay` → `Guided, with trail`.

## 7. Размеры задачи и пределы верификации

Для **симуляции** годится почти любой `N` (упирается в скорость, а не в
память).

Для **верификации** Spin обходит весь граф состояний. Размер графа
растёт **экспоненциально** от числа параллельных процессов и от
размера локальных массивов. Практическая граница — около `N ≈ 4–5`:
дальше упирается в локальные `A, B` у каждого из `P` работников
(state vector ≈ `2·P·N²` байт только за счёт них).

При `N = 10` и больше — `pan` упрётся в `MEMLIM` (`Warning: Search
not completed`). Решения:

1. **Для верификации временно поставить `N = 3` или `N = 4`.**
   Корректность алгоритма не зависит от размера данных.
2. Перекомпилировать `pan` со сжатием:
   ```bash
   gcc -DCOLLAPSE -O2 -o pan pan.c
   ```
3. Поднять лимит памяти:
   ```bash
   gcc -DMEMLIM=4096 -O2 -o pan pan.c
   ```
   В iSpin: `Show Advanced Parameter Settings` → memory limit.

## 8. Изменение `N` и `P`

В начале `matmul2.pml`:

```c
#define N    3   /* размер матриц N × N        */
#define P    3   /* число рабочих процессов    */
```

Алгоритм работает **для любых `N ≥ 1` и `P ≥ 1`**:

- `P > N` — лишние работники блокируются на `task[id] ?` и в конце
  получают `STOP`.
- `P < N` — каждый работник обрабатывает несколько столбцов
  последовательно (по мере освобождения, динамическая балансировка).
- `P = N` — равномерное распределение, по одному столбцу на работника.

## 9. Известные «гремлины» и как их распознать

| Симптом                                                       | Причина                                          | Лечение                                  |
|---------------------------------------------------------------|--------------------------------------------------|------------------------------------------|
| `couldn't find claim 1`, `trail ends after -N steps`          | Старый/сломанный `.pml.trail` от прошлого прогона | `rm matmul2.pml.trail` или Mode → Random |
| `Warning: Search not completed`, `reached -DMEMLIM bound`     | Слишком большой `N` для верификации              | Уменьшить `N` или включить `-DCOLLAPSE`  |
| `errors: 1, 0 states stored`                                  | `pan` упал на инициализации (часто из-за памяти) | То же — это не «нашёл ошибку»            |
| `acceptance cycles + (fairness disabled)` + counterexample    | Запустил liveness без `-f`/galки fairness        | Включить weak fairness                   |
| `couldn't open "Worker"`                                      | iSpin GUI ищет отдельный файл для proctype       | Безвредно, игнорировать                  |
| Verification идёт без LTL (`-DNOCLAIM`, `never claim - (not selected)`) | На вкладке Verification стоит `do not use a never claim` | Поставить `use claim` + имя свойства     |
